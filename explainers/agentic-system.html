What is an Agentic System?

The full spectrum of AI autonomy — from single-turn responses to self-directing networks — and how to design them safely.

An agentic system is an AI architecture where one or more models perceive their environment, plan actions, use tools, and iterate toward a goal — with varying degrees of autonomy and human involvement.

The Key Property

Agenticity isn't binary — it's a spectrum from a single prompt-response to a fully autonomous multi-agent network. The key variable is how much the system decides for itself.

Why It Matters Now

As LLMs become more capable, the practical limit on what agents can do expands. What required a human operator in 2022 can now run autonomously at scale.

Higher autonomy = higher leverage and higher risk. The design question isn't 'how autonomous can we make this?' but 'what level of autonomy matches the stakes of this task?'

L0: Simple Responder

A model that responds to a single prompt with no tools, no memory, no follow-up.

Example: "What's the capital of France?" → "Paris"

Autonomy
None
Human
100% of decisions
Risk
Very Low

L1: Tool-Augmented

A model that can call a predefined set of tools (search, calculator) within a single response turn.

Example: Weather query → calls weather API → returns current temp

Autonomy
Minimal
Human
All goals set
Risk
Low

L2: Workflow Agent

An agent that executes a multi-step workflow, using tools across several loops to complete a defined task.

Example: "Write me a report on X" → searches, reads, synthesises, formats

Autonomy
Moderate
Human
Sets goal; chooses steps
Risk
Medium

L3: Autonomous Agent

An agent that can set sub-goals, handle unexpected states, and operate without human check-ins for extended periods.

Example: "Monitor this data feed and alert me if anomaly detected"

Autonomy
High
Human
Sets objective only
Risk
High

L4: Multi-Agent

A network of specialised agents that coordinate — one plans, others execute, one validates — all working toward a shared goal.

Example: Orchestrator routes tasks to code agent, research agent, and writer agent

Autonomy
Very High
Human
Approves output
Risk
Very High

L5: Self-Improving

A system that can modify its own prompts, tools, or sub-agents based on performance — learning from outcomes.

Example: Agent notices its research is poor → rewrites its own search instructions

Autonomy
Full
Human
Oversight only
Risk
Critical

Core Model

The LLM at the centre — reasons, plans, generates text, decides which tools to call.

Role: Brain / Reasoner

Tool Layer

The set of APIs, functions, and capabilities the agent can call — web search, code execution, file I/O.

Role: Hands / Actuators

Memory Layer

In-context conversation + external retrieval (RAG) + episodic storage for persistence across sessions.

Role: Memory / State

Orchestrator

Logic that decides when to loop, when to stop, how to handle errors, when to ask for human input.

Role: Director / Control flow

Execution Sandbox

An isolated environment where code runs safely — contained from the host system to limit blast radius.

Role: Safety / Containment
High ▼
Unintended Actions
An agent with write access can create, overwrite, or delete real files and send real messages. Mistakes have real consequences. Mitigation: Use read-only tools where possible; require confirmation for writes
Medium ▼
Prompt Injection
Malicious content in tool outputs (web pages, documents) can hijack agent behaviour by embedding instructions. Mitigation: Treat all external content as untrusted data; validate before acting
High ▼
Goal Misalignment
The agent optimises for the stated goal in unexpected ways — technically correct but not what the human meant. Mitigation: Specify constraints, not just objectives; add human review checkpoints
Medium ▼
Runaway Loops
Without a termination condition, agents can loop indefinitely, consuming resources or performing repeated actions. Mitigation: Set explicit step limits and timeout conditions
Medium ▼
Scope Creep
An autonomous agent may access or modify things outside its intended scope when pursuing a goal. Mitigation: Apply least-privilege: minimum tool access needed for the task
